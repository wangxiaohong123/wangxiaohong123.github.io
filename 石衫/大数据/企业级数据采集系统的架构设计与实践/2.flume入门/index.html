数据流动模型

source是可以通过监听http端口来接收数据，也可以通过读取本地磁盘，或者自己写一个source等等，source收到数据后把数据交给channel，channel可以把数据写到磁盘，也可以写到内存，sink从channel读取数据可以发送给kafka或者hdfs或者其他flume都可以。

flume怎么保证数据的不丢失

在source收到数据后交给channel，channel把数据写到本地磁盘或者内存的时候才算写入成功，如果写入的是磁盘，这个时候消息时不会丢失的，当source返回成功一定是成功，如果返回失败，nginx收到http的异常码会进行故障转移，发送给另一个flume。当sink把数据存到kafka或者其他组件的时候才会删除channel的数据，如果channel是纯内存的话，宕机的时候就可能会丢掉数据。

配置

一些配置很简单，直接去官网看就可以http://flume.apache.org/releases/content/1.9.0/FlumeUserGuide.html

fan-out（扇出）

flume经常会出现一个source对应多个channel+sink的情况，source收到数据之后可以有两种模式交给不同的channel，第一种就是replicating，类似广播，所有channel都能收到这个数据，第二种就是multiplexing，会根据head中的参数进行路由，head的参数可以在拦截器中插入，用的不多。

常用source

  • Avro Source：常用于两个agnet之间的数据流转。
  • Spooling Directory Source：这个一般用于获取用户的行为日志，比如部署在tomcat里的web项目的日志直接写入本地磁盘，就得用这个方式获取日志。
  • Kafka Source：让flume从kafka里消费数据。
  • HTTP Source：监听http端口。

常用channel

  • Memory Channel：不写磁盘，直接基于内存队列，skin从内存队列消费数据。
  • File Channel：写磁盘，性能慢，但是数据不会丢。
  • Spillable Memory Channel：优先写内存队列，如果内存队列慢了就写磁盘。

常用sink

  • HDFS Sink
  • Hive Sink
  • Avro Sink
  • Kafka Sink

Flume Sink Processors

这个就是把多个sink变成一个sink，比如一个channel对应多个sink，这个时候channel的数据是负载均衡（load_balance）给到对应的sink，也可以做故障转移（fail over），比如一个agent直连3个agent，那么就可以配置多个sink，每个sink对应一个直连的agent，这个时候只要processors设置成load_balance就可以实现负载均衡，就高可用了。

拦截器

Timestamp Interceptor、Host Interceptor、Static Interceptor、Remove Header Interceptor（删除用的）、UUID Interceptor、Morphline Interceptor、Regex Filtering Interceptor等等吧，用的不多，了解就行。

jvm参数

在flume文件夹下的conf/flume-env.sh配置。

日志

日志的配置文件也在conf下。
