2014年开始起步，那时候国内做大数据的只有少数的互联网公司，用什么MapReduce手写作业，然后flume抽取数据，生成报表写入MySQL，也没有数仓的概念。

2015年开始有人用spark、storm，当时就干一件事，出报表。

2016年spark爆火。

2017年spark SQL替代hive QL来做数仓的ETL开发，spark streaming替代storm，这一年spark好牛逼。

这个时候如果想看数据结构，使用的是presto或者impala。

然后使用kylin进行OLAP分析，比如说几亿条数据预聚合成几千万条，然后使用kylin根据不同的维度把几千万条聚合成几十条这样的，联机分析处理。

druid是真对实时的数据的聚合分析统计，实时OLAP。

elasticSearch elk，简单的分析，代替SQL。

hbase，联机在线数据查询。

YARN，资源调度系统，还有Oozie、Azkaban、宙斯，这种东西是和离线处理结合起来使用的。

目前的广泛的应用体系：

离线：flume/canal -> HDFS -> Hive -> presto/Impala -> 数据仓库/数据治理 -> Spark SQL -> 作业调度 -> YARN -> ES -> 报表系统

实时：flume/canal -> kafka -> Storm/Spark Streaming -> HBase -> 实时数据大盘

2018年，flink出来替代Storm/Spark Streaming，处理的东西一样。

大公司的大数据职位分工

数据平台工程师：

这个是一个几百人的大团队，拆分成很多中型团队，有的团队负责离线，有的团队负责实时，每个中型团队有很多小团队，有的小团队专门负责维护HBase这样。每个小团队负责集群部署、集群监控、日常运维、数据检查、版本升级、数据备份、可视化工作台开发、故障处理、业务团队使用的支撑、二次开发、阅读源码、贡献社区等等。

数据仓库工程师：进行数据仓库的建模、分层、定义指标，开发ETL，跑数据，就是建模+SQL。

数据应用工程师：大数据技术的使用，比如业务项目（2C的业务流程），或者基础性的（BI系统）。

AI/机器学习/数据挖掘工程师：一些需要智能化的产品，比如外卖调度平台，数据仓库工程师出数据，数据应用工程师开发系统，算法工程师出算法最后结合起来，但是现在大部分都是在冥思苦想，到底做出什么东西才能对业务产生价值。

风控工程师、反欺诈工程师、推荐工程师、搜索工程师：特定领域的算法、系统、数据，都是他们自己来做。
